# Fused MoE 多智能体优化工作流

更新日期：2026-07-13

## 1. 目标与基本原则

本工作流用于后续 Fused MoE TileLang 算子优化。采用“常驻 XHigh 主智能体、Max 探索、两个 Medium 双机实验、按需 Ultra 复盘”的组织方式，在最多 4 个并发槽位内兼顾探索深度、实验速度和版本安全。

核心原则：

- 主智能体拥有正式版本和最终决策权。
- 探索智能体只读，不修改代码、服务器或 OJ 状态。
- 两个实验智能体分别独占一台服务器，候选文件相互隔离。
- Ultra 只用于重大路线选择、复杂异常归因和阶段复盘。
- OJ 默认由主智能体提交，不长期占用独立并发槽位。
- 成功、退化、编译失败和平台异常都必须记录到 `docs/`。

## 2. 角色配置

| 角色 | 模型与推理档位 | 主要职责 |
|---|---|---|
| 主智能体 | GPT-5.6 Sol，XHigh | 调度、实验选择、版本管理、结果归因、交叉验证、OJ 提交 |
| 探索智能体 | GPT-5.6 Sol，Max | 只读分析瓶颈，提出结构方向、风险和最小实验矩阵 |
| 实验智能体 A | GPT-5.6 Sol，Medium | 独占服务器-64-1，执行候选 A |
| 实验智能体 B | GPT-5.6 Sol，Medium | 独占服务器-64-2，执行候选 B |
| 阶段复盘智能体 | GPT-5.6 Sol，Ultra | 按需替换已结束的探索槽位，处理重大决策和复杂归因 |

峰值并发为主智能体加三个子智能体，共 4 个槽位。

## 3. 标准优化循环

### 3.1 建立不可变基线

主智能体在每轮开始前固定：

- 当前正式 `submission.py`。
- 代码哈希或可唯一识别的版本号。
- 最近一次 Accepted 的 OJ 提交号、总分和三个分点耗时。
- 两台服务器上的基线耗时。
- 可立即恢复的备份版本。

正式基线在本轮实验完成前不得被实验智能体覆盖。

### 3.2 Max 探索

探索智能体只接收必要上下文：

- 当前 kernel 结构。
- 正式测试 shape。
- 当前 OJ 分数与耗时。
- 已验证保留项和明确淘汰项。
- 硬件与 TileLang 约束。

探索报告必须包含：

1. 主要瓶颈。
2. 至少六个互不重复的新方向。
3. 每个方向的机制、预期收益、正确性风险和编译风险。
4. 最小验证实验。
5. Top 3 优先级。
6. 两台服务器可并行执行的实验矩阵。

探索智能体不得修改文件、连接服务器或提交 OJ。

### 3.3 主智能体选择实验

主智能体每轮优先选择两个相互独立、能够解释收益来源的单变量实验。每个实验分配唯一 ID，例如：

```text
R47-down-threads256
R48-down-swizzle8
```

实验任务必须明确：

- 基线版本或哈希。
- 唯一允许改变的变量。
- 分配的服务器。
- 正确性用例、正式 shape、warmup、iters 和 seed。
- 相对提升的比较基准。
- 成功、淘汰和停止条件。

### 3.4 两个 Medium 并行实验

实验智能体 A 只操作服务器-64-1，实验智能体 B 只操作服务器-64-2。

每个实验智能体只能：

- 修改自己的候选副本。
- 上传到分配的服务器。
- 执行编译、allclose、性能和资源测试。
- 返回标准化实验结果。

禁止：

- 直接覆盖本地正式 `submission.py`。
- 操作另一台服务器。
- 提交 OJ。
- 将未经验证的候选宣布为正式最佳。

推荐结果格式：

```json
{
  "experiment_id": "R47-down-threads256",
  "base_version": "52062",
  "server": "server-64-1",
  "correct": true,
  "point_ms": [8.0, 14.0, 26.0],
  "baseline_ms": [8.0, 14.0, 26.0],
  "improvement_pct": [0.0, 0.0, 0.0],
  "compile_error": null,
  "resource_notes": null,
  "recommendation": "keep_or_reject"
}
```

## 4. 候选晋级门槛

候选出现以下任一情况时直接淘汰：

- 编译失败且不存在明确可修正的实现错误。
- allclose 或正式正确性检查失败。
- 任一正式点明显退化，且没有足够的总分收益补偿。
- 性能提升无法在复测中重现。
- 资源占用、显存或 shared memory 超过比赛限制。

候选进入交叉验证至少需要：

- 所有已测 shape 正确。
- 至少一个正式点有明确提升。
- 其他正式点没有明显退化。
- 改动机制能够解释观测结果。

正式提交 OJ 前至少满足：

- 两台服务器均 allclose。
- 关键点跨服务器差异原则上不超过约 1%。
- Pareto 前沿候选长测至少两次。
- 本地正式候选文件与远端测试文件一致。
- 主智能体再次核对关键参数、文件首尾和代码哈希。

## 5. Ultra 调用条件

日常调度由 XHigh 主智能体完成。只有出现以下情况时，才结束或暂停 Max 探索任务，并在空出的槽位调用 Ultra 复盘智能体：

- 两台服务器的性能趋势相互冲突。
- OJ 与服务器结果显著不一致。
- 连续多轮没有提升，需要更换路线。
- 需要在多个结构性方向间做阶段选择。
- 出现难以解释的正确性、编译器或资源异常。
- 准备进行大规模 kernel 重构。

Ultra 复盘输出应以决策为中心，不承担上传、等待、批量测试或日志整理。

## 6. OJ 提交流程

OJ 默认由主智能体提交。独立 OJ Medium 角色仅在实验任务结束、有空闲槽位时临时启用，并且只能读取指定的不可变候选，不能修改代码。

提交步骤：

1. 确认候选版本号或哈希。
2. 确认两台服务器的正确性和复测结果。
3. 检查 `submission.py` 首尾完整。
4. 检查所有关键参数和 shape 分派。
5. 在 OJ 编辑器中搜索关键标志，确认代码粘贴完整。
6. 提交并记录提交号。
7. 读取总分、总耗时和三个分点结果。
8. 若失败，必须展开 stderr，区分代码错误、超时、编译失败和评测机 OOM。

评测机 OOM、队列异常或平台故障不得直接归因为 kernel 错误。原样重提时必须记录原因和新提交号。

## 7. 文件所有权与隔离

- `tilelang-metax/race_tests/moe/submission.py`：只有主智能体可晋级和修改正式版本。
- `docs/moe_optimization_log.md`：主智能体维护最终实验账本。
- 探索智能体：对工作区完全只读。
- 实验智能体：只编辑自己的候选文件或实验目录。
- 服务器-64-1：只由实验智能体 A 操作。
- 服务器-64-2：只由实验智能体 B 操作。

推荐候选目录结构：

```text
experiments/
  R47-down-threads256/
    submission.py
    result.json
  R48-down-swizzle8/
    submission.py
    result.json
```

正式文件只在候选通过晋级门槛后，由主智能体使用明确补丁更新。

## 8. 实验日志要求

每轮必须记录：

- 轮次和实验 ID。
- 改动内容与实验假设。
- 使用的服务器和代码版本。
- 正确性结果。
- 三个正式点耗时。
- 相对当前基线的提升百分比。
- 编译、寄存器、shared memory、显存和异常信息。
- 保留、淘汰或继续复测的结论。
- OJ 提交号和分点结果（若提交）。

不得省略失败实验。失败结果用于避免重复探索同一无效方向。

## 9. 推荐调度节奏

一个标准阶段建议按以下节奏运行：

1. Max 生成结构方向和实验队列。
2. 主智能体选择两个单变量实验。
3. 两个 Medium 双机并行执行。
4. 主智能体筛选 Pareto 前沿。
5. 胜出候选跨服务器复现。
6. 必要时调用 Ultra 做阶段复盘。
7. 主智能体提交 OJ。
8. 固化正式版本、日志和下一轮队列。

该循环持续进行，直到达到比赛目标、可用方向耗尽，或新的实验成本明显高于预期收益。
